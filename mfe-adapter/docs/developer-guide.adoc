== Purpose

This guide explains how to use the MFE Adapter to create a new, production-ready MFE Adapter service.
It is aimed at backend developers who are comfortable with Spring Boot and Gradle.
No changes to the template source are required for the common use case; customisation is done entirely through configuration and SPI implementations.

== Prerequisites

Before working with this template, ensure the following tools are installed:

* JDK 21 or later (`java -version` should report `21.x.x`).
* Gradle 9.4.1 — the Gradle wrapper included in the project handles this automatically.
* Docker (for running the OIDC Provider, Kafka, and the target microservice locally).
* Access to a Maven-compatible artifact repository hosting Spring Boot 4 artifacts.

== Getting Started

=== Clone the Template

[source,bash]
----
git clone <this-repo-url> my-mfe-adapter
cd my-mfe-adapter/mfe-adapter
----

=== Generate the Gradle Wrapper Binary

The wrapper JAR is not committed to source control.
Run the following command once to generate it:

[source,bash]
----
gradle wrapper --gradle-version 9.4.1
----

=== Run the Tests

[source,bash]
----
./gradlew test
----

All unit tests should pass out of the box.
The persistence adapter tests use the `test` Spring profile which activates an in-memory H2 database automatically.

=== Start the Application Locally

Start a local OIDC Provider and Kafka instance using Docker Compose (see <<deployment-guide.adoc#local-docker-compose,Deployment Guide — Local Docker Compose>>), then run:

[source,bash]
----
./gradlew bootRun --args='--spring.profiles.active=local'
----

The MFE Adapter starts on port 8080 with mTLS disabled.
The H2 console is available at http://localhost:8080/h2-console.

== Simple Forwarding (No Customisation Required)

The most common MFE Adapter use case is pure forwarding: the MFE Adapter verifies that the API gateway validated the API key, exchanges the session cookie for a user token, exchanges the user token for an inner service token, and forwards the request to the microservice unchanged.
No custom code is required for this scenario.
The template handles all three steps automatically.

TIP: *No coding required.* +
To use the MFE Adapter in simple-forwarding mode you only need to supply the correct configuration values.
You do not write, modify, or register any Java classes.
Skip straight to <<_required_configuration>> and fill in the YAML properties for your environment.

=== How It Works

The following sequence diagram shows the complete flow for the simple forwarding case.

.Simple forwarding sequence
plantuml::design/diagrams/sequence-simple-forwarding.puml[format=svg,opts=inline]

When a request arrives from the MFE the following steps occur in order:

. *API key check* — `ApiKeyValidationFilter` delegates to an injected `ApiKeyValidator` bean.
If no `ApiKeyValidator` bean is declared the filter is a no-op and all requests pass through.
When `StandardApiKeyValidator` is configured it rejects any request that does not carry the `X-API-Key-Validated: true` header (or whichever header and value are configured under `mfe-adapter.api-gateway.*`), returning HTTP 403.
See <<_api_key_validation>> for how to enable this layer.
. *Session resolution* — The session cookie is read, the corresponding session record is looked up in H2, and the stored user token is retrieved.
  If the cookie is missing or the session has expired, the MFE Adapter returns HTTP 401.
. *Inner token exchange* — When `mfe-adapter.authorization-service.required=true` (the default), the user token is sent to the Authorization Service at `mfe-adapter.authorization-service.base-url` + `mfe-adapter.authorization-service.token-swap-path`.
  The Authorization Service returns a short-lived inner token scoped to the target microservice.
  Set `mfe-adapter.authorization-service.required=false` to skip this step and run in pure session-authenticated proxy mode.
. *Forwarding* — The original request headers, query string, and body are forwarded to the microservice at `mfe-adapter.microservice.base-url`, with the `Authorization` header replaced by the inner token.
  The microservice response (status, headers, body) is returned to the MFE as-is.

Because no `RequestValidator`, `RequestTransformer`, or `ResponseTransformer` beans are registered, the pipeline performs no additional validation or rewriting.

=== Required Configuration

Set the following properties (via environment variables in production, or `application-local.yml` for local development):

[source,yaml]
----
mfe-adapter:
  oidc-provider:
    issuer-uri: ${MFE_ADAPTER_OIDC_PROVIDER_ISSUER_URI}
    realm: ${MFE_ADAPTER_OIDC_PROVIDER_REALM}
    client-id: ${MFE_ADAPTER_OIDC_PROVIDER_CLIENT_ID}
    client-secret: ${MFE_ADAPTER_OIDC_PROVIDER_CLIENT_SECRET}

  authorization-service:
    base-url: ${MFE_ADAPTER_AUTHZ_BASE_URL}
    # token-swap-path defaults to /token/swap

  microservice:
    base-url: ${MFE_ADAPTER_MS_BASE_URL}
    # context-path defaults to /api
----

No SPI beans need to be defined.
The default behaviour of `MfeAdapterRequestApplicationService` is to forward the request unchanged once the token exchange completes.

NOTE: The six properties above (OIDC Provider, Authorization Service, and microservice coordinates) are the *only* changes needed to put the MFE Adapter into production in simple-forwarding mode.
There is no application code to write.

=== Verifying the Setup

After starting the stack with Docker Compose and running the MFE Adapter with the `local` profile, send a request with an API-key-validated header:

[source,bash]
----
curl -v \
  -H "X-API-Key-Validated: true" \
  -H "Cookie: MFESESSION=<session-id>" \
  http://localhost:8080/api/your-endpoint
----

A successful response from the microservice confirms that all three steps (API key check, session resolution, inner token exchange) completed successfully.

[[_api_key_validation]]
=== Enabling API-Key Validation

By default `ApiKeyValidationFilter` is a no-op: no `ApiKeyValidator` bean is registered and all requests pass through without inspection.
To activate the standard AG header check, declare `StandardApiKeyValidator` as a Spring bean in your application configuration:

[source,java]
----
@Bean
public ApiKeyValidator apiKeyValidator(MfeAdapterProperties properties) {
    return new StandardApiKeyValidator(properties);
}
----

The two properties that control which header and value the `StandardApiKeyValidator` inspects are:

[source,yaml]
----
mfe-adapter:
  api-gateway:
    validated-header: X-API-Key-Validated   # header set by the AG
    validated-value:  "true"                 # expected value
----

You can supply a completely custom implementation by declaring any bean that implements `ApiKeyValidator`.

=== Enabling API-Key Scope Forwarding

When `StandardApiKeyValidator` (or any custom `ApiKeyValidator`) is active and returns a scope, the MFA can forward the resolved scope to the microservice.
Scope forwarding is enabled by default (`mfe-adapter.api-gateway.forward-scope=true`).

[source,yaml]
----
mfe-adapter:
  api-gateway:
    scope-header: X-API-Key-Scope   # header forwarded to the MS
    forward-scope: true              # set to false to suppress forwarding
----

The scope value (`READ`, `WRITE`, or `AUDIT`) is also available to SPI implementations via `ProxiedRequest.getApiKeyScope()`.

=== Enabling Version Extraction

Declare a `VersionExtractionStrategy` bean to extract the API version from inbound requests.

Two ready-made implementations are provided:

`PathVersionExtractionStrategy` — parses the version from a `/v{n}/` path prefix (optionally followed by a channel segment):

[source,java]
----
@Bean
public VersionExtractionStrategy versionExtractionStrategy() {
    return new PathVersionExtractionStrategy();
}
----

`HeaderVersionExtractionStrategy` — reads the version from a configurable header:

[source,java]
----
@Bean
public VersionExtractionStrategy versionExtractionStrategy(MfeAdapterProperties props) {
    return new HeaderVersionExtractionStrategy(props);
}
----

[source,yaml]
----
mfe-adapter:
  versioning:
    version-header: X-API-Version   # header carrying the version
----

The extracted version is stored in `ProxiedRequest.getVersion()` and is available to SPI implementations.
The path prefix is stripped before forwarding when using `PathVersionExtractionStrategy`.

=== Enabling Custom Routing

Declare a `RoutingStrategy` bean to control how inbound paths are translated before forwarding to the microservice.

Three ready-made implementations are provided:

`PassThroughRoutingStrategy` (default, used automatically when no bean is declared) — forwards path unchanged.

`PrefixStrippingRoutingStrategy` — strips the microservice context-path prefix:

[source,java]
----
@Bean
public RoutingStrategy routingStrategy(MfeAdapterProperties props) {
    return new PrefixStrippingRoutingStrategy(props);
}
----

`PathMappingRoutingStrategy` — translates path prefixes via a static mapping table:

[source,java]
----
@Bean
public RoutingStrategy routingStrategy(MfeAdapterProperties props) {
    return new PathMappingRoutingStrategy(props);
}
----

[source,yaml]
----
mfe-adapter:
  routing:
    path-mappings:
      "/api/v1": "/api"
      "/api/v2": "/v2/api"
----

== Customising the MFE Adapter

All customisation is done by implementing one or more SPI interfaces and registering the implementations as Spring beans.
The template automatically discovers all beans that implement the SPI interfaces via Spring dependency injection.
No changes to the template code are required.

=== Request Flow With SPI Implementations

The following sequence diagram shows how the pipeline changes when one or more SPI beans are registered.
Compared to the simple forwarding case, the `MfeAdapterRequestApplicationService` invokes each registered `RequestValidator` in turn before forwarding, then each `RequestTransformer` to rewrite the outbound request, and finally each `ResponseTransformer` to rewrite the response before returning it to the MFE.

=== SPI Interfaces

Three extension point interfaces are provided in the `com.arc_e_tect.book.sedr.mfeadapter.domain.spi` package.

==== `RequestValidator`

Implement `RequestValidator` to add business or content validation rules.
If the validator returns a failure result, the MFE Adapter returns HTTP 422 to the MFE without forwarding the request to the MS.

[source,java]
----
@Component
public class CountryCodeValidator implements RequestValidator {

    private final ReferenceDataStorePort refDataStore;

    public CountryCodeValidator(ReferenceDataStorePort refDataStore) {
        this.refDataStore = refDataStore;
    }

    @Override
    public ValidationResult validate(ProxiedRequest request) {
        // Parse body, extract country code, look it up in the cache
        String countryCode = extractCountryCode(request);
        return refDataStore.findByTypeAndCode("country-codes", countryCode)
                .map(e -> ValidationResult.ok())
                .orElse(ValidationResult.failure("Unknown country code: " + countryCode));
    }
}
----

Multiple `RequestValidator` beans are all invoked in order.
If any validator returns a failure, the request is rejected.

==== `RequestTransformer`

Implement `RequestTransformer` to rewrite the proxied request before it is forwarded to the MS.
This is useful for path rewriting, adding custom headers, or replacing request body fields.

[source,java]
----
@Component
public class VersionPrefixTransformer implements RequestTransformer {

    @Override
    public ProxiedRequest transform(ProxiedRequest request) {
        String rewrittenPath = "/v2" + request.getPath();
        return ProxiedRequest.builder()
                .method(request.getMethod())
                .path(rewrittenPath)
                .queryString(request.getQueryString())
                .headers(request.getHeaders())
                .body(request.getBody())
                .sessionId(request.getSessionId())
                .build();
    }
}
----

Multiple `RequestTransformer` beans are applied in Spring bean order.
Each transformer receives the output of the previous one.

==== `ResponseTransformer`

Implement `ResponseTransformer` to rewrite the MS response before it is returned to the MFE.
Common uses include field masking, response body enrichment, and content-type normalisation.

[source,java]
----
@Component
public class SensitiveFieldMaskTransformer implements ResponseTransformer {

    @Override
    public ProxiedResponse transform(ProxiedResponse response, ProxiedRequest request) {
        byte[] masked = maskSensitiveFields(response.getBody());
        return ProxiedResponse.builder()
                .statusCode(response.getStatusCode())
                .headers(response.getHeaders())
                .body(masked)
                .build();
    }
}
----

=== Consuming the Reference-Data Cache

SPI implementations can inject `ReferenceDataStorePort` to query the local cache without making a synchronous HTTP call:

[source,java]
----
@Component
public class CurrencyEnricher implements RequestTransformer {

    private final ReferenceDataStorePort refDataStore;

    public CurrencyEnricher(ReferenceDataStorePort refDataStore) {
        this.refDataStore = refDataStore;
    }

    @Override
    public ProxiedRequest transform(ProxiedRequest request) {
        // Look up cached currency metadata and add as a request header
        refDataStore.findByTypeAndCode("currencies", "EUR")
                .ifPresent(entry -> request.getHeaders()
                        .computeIfAbsent("X-Currency-Name", k -> List.of())
                        // headers are immutable in ProxiedRequest; use the builder instead
                        );
        return request; // return a modified copy in real code
    }
}
----

== Configuration Reference

All MFE Adapter-specific settings live under the `mfe-adapter.*` namespace in `application.yml`.
The following table lists every supported property.

[cols="2,1,3"]
|===
| Property | Default | Description

| `mfe-adapter.session.cookie-name`
| `MFESESSION`
| Name of the session cookie returned to the MFE.

| `mfe-adapter.session.ttl-minutes`
| `30`
| Session time-to-live in minutes.

| `mfe-adapter.session.http-only`
| `true`
| If `true`, the cookie is not accessible via JavaScript.

| `mfe-adapter.session.secure`
| `true`
| If `true`, the cookie is only sent over HTTPS.
  Set to `false` in the `local` profile.

| `mfe-adapter.session.same-site`
| `Strict`
| `SameSite` cookie attribute.

| `mfe-adapter.session.cleanup-interval-ms`
| `300000`
| Interval (ms) between scheduled H2 session-expiry purges.

| `mfe-adapter.api-gateway.validated-header`
| `X-API-Key-Validated`
| Header name set by the AG after API-key validation.

| `mfe-adapter.api-gateway.validated-value`
| `true`
| Expected value of the validation header.

| `mfe-adapter.api-gateway.scope-header`
| `X-API-Key-Scope`
| Header name used to carry the API-key scope both from the AG and when forwarding to the MS.

| `mfe-adapter.api-gateway.forward-scope`
| `true`
| When `true`, the resolved scope is forwarded to the MS under `scope-header`.
  Set to `false` to suppress scope forwarding.

| `mfe-adapter.oidc-provider.issuer-uri`
| —
| OIDC Provider issuer URL.
  Example: `https://idp.example.com`.

| `mfe-adapter.oidc-provider.realm`
| —
| Realm name (provider-specific; leave blank if not applicable).

| `mfe-adapter.oidc-provider.client-id`
| —
| OAuth2 client identifier registered in the OIDC Provider.

| `mfe-adapter.oidc-provider.client-secret`
| —
| OAuth2 client secret.
  Supply via environment variable `MFE_ADAPTER_OIDC_PROVIDER_CLIENT_SECRET`.

| `mfe-adapter.authorization-service.base-url`
| —
| Base URL of the Authorization Service.

| `mfe-adapter.authorization-service.token-swap-path`
| `/token/swap`
| Path of the token-swap endpoint relative to `mfe-adapter.authorization-service.base-url`.

| `mfe-adapter.authorization-service.required`
| `true`
| When `true` (the default), the SecService token-swap step is performed for every proxied request.
  Set to `false` to skip inner-token acquisition and run in pure session-authenticated proxy mode.

| `mfe-adapter.microservice.base-url`
| —
| Base URL of the target microservice.

| `mfe-adapter.microservice.context-path`
| `/api`
| Context path prefix used by `PrefixStrippingRoutingStrategy`.

| `mfe-adapter.versioning.version-header`
| `X-API-Version`
| Header read by `HeaderVersionExtractionStrategy` when extracting the API version.

| `mfe-adapter.routing.strip-prefix`
| `false`
| Shorthand flag; set to `true` as an alternative to declaring `PrefixStrippingRoutingStrategy` explicitly.

| `mfe-adapter.routing.path-mappings`
| _(empty)_
| Ordered map of source-prefix → target-prefix entries used by `PathMappingRoutingStrategy`.

| `mfe-adapter.mtls.enabled`
| `false`
| Enable mTLS for outbound `WebClient` connections.
  Set to `true` in production and supply keystore/truststore paths.

| `mfe-adapter.mtls.key-store`
| —
| Path or classpath reference to the PKCS#12 keystore.

| `mfe-adapter.mtls.key-store-password`
| —
| Keystore password.
  Supply via environment variable `MFE_ADAPTER_KEYSTORE_PASSWORD`.

| `mfe-adapter.mtls.trust-store`
| —
| Path or classpath reference to the PKCS#12 truststore.

| `mfe-adapter.mtls.trust-store-password`
| —
| Truststore password.
  Supply via environment variable `MFE_ADAPTER_TRUSTSTORE_PASSWORD`.

| `mfe-adapter.kafka.reference-data.topic-pattern`
| `reference-data\\..*`
| Java regex matched against Kafka topic names.
  All matching topics are consumed.

| `mfe-adapter.kafka.consumer.group-id`
| `mfe-adapter-reference-data`
| Kafka consumer group identifier.
|===

== Adding New Unit Tests

New tests live under `src/test/java/com/arc_e_tect/book/sedr/mfeadapter/` and should be annotated with `@ActiveProfiles("test")` when they need Spring context.
Pure Mockito unit tests (no Spring context) run faster and should be preferred for application service and domain logic testing.
Use plain Mockito unit tests (`@ExtendWith(MockitoExtension.class)`) for persistence adapter tests — mock the Spring Data repository and use `@InjectMocks` for the adapter.
`@DataJpaTest` is not available in Spring Boot 4.
