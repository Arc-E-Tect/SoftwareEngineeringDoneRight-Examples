== ADR Summary

=== ADR-001: Adoption of the Micro-Frontend Adapter Pattern

*Status:* Accepted

==== Context

We require an architecture that:

- Supports incremental and iterative delivery
- Preserves DDD bounded contexts
- Maintains clean ubiquitous language
- Enables independent evolution of MFEs and MSs
- Avoids cross-context coupling
- Aligns with Hexagonal Architecture
- Supports BDD practices
- Supports multiple frontend channels (web, mobile, agentic)

The traditional BFF pattern aggregates multiple microservices and introduces cross-context coupling, violating these principles.

==== Decision

We adopt the *Micro-Frontend Adapter (MFA)* pattern with explicit cardinalities:

- Each MFE associates with one and only one MFA.
- Each MFA associates with one and only one MFE.
- Each MFA associates with one and only one MS.
- Each MS can associate with one or more MFAs.

Additional constraints:

- MFA forms part of the Frontend ACL.
- MFA handles boundary concerns only.
- MFA mirrors Hexagonal Architecture.
- No aggregation or orchestration across MSs is allowed.
- No MFE or MFA spans multiple bounded contexts.

==== Consequences

Positive:

- Clean bounded contexts and ubiquitous languages
- Independent delivery of MFEs and MSs
- Maintainable long-term architecture
- Clear testability and BDD alignment
- Reduced coupling and complexity
- Clear, auditable boundaries and data flows

Negative:

- More deployable components than a single BFF
- Requires discipline to maintain boundaries

These trade-offs are acceptable and aligned with long-term architectural goals.
