== Overview

This document defines the architectural approach for structuring the frontend and backend of our system using:

- Micro-Frontends (MFEs)
- Micro-Frontend Adapters (MFAs)
- Microservices (MSs)
- Hexagonal Architecture (Ports & Adapters)
- Domain-Driven Design (DDD)
- Behavior-Driven Development (BDD)
- An explicit Frontend Anti-Corruption Layer (ACL)

The goal is to create an architecture that:

- Supports **incremental and iterative delivery** of features
- Preserves **bounded contexts** and **ubiquitous language**
- Enables **independent evolution** of MFEs and MSs
- Ensures **maintainability** and **testability**
- Avoids the long-term complexity and coupling risks of the traditional BFF pattern

This document is intended for:

- Software engineers and test engineers
- Architects and CTOs
- Internal and external IT auditors
- Stakeholders evaluating long-term maintainability and scalability

== Architectural Summary

The architecture consists of three primary layers:

1. **Micro-Frontend (MFE)** — the UI for a single bounded context.
2. **Micro-Frontend Adapter (MFA)** — a thin, hexagonal adapter forming the Frontend ACL.
3. **Microservice (MS)** — the backend domain service for the same bounded context.

=== Relationship Cardinalities

The relationships are **explicit and constrained**:

- Each **MFE associates with one and only one MFA**.
- Each **MFA associates with one and only one MFE**.
- Each **MFA associates with one and only one MS**.
- Each **MS can associate with one or more MFAs**.

This supports multiple frontend types (e.g. web, mobile, agentic) each with:

- Its own MFE
- Its own MFA
- All targeting the **same** MS (same bounded context, different channels)

In short:

- `MFE 1..1 ↔ 1..1 MFA`
- `MFA 1..1 → 1..1 MS`
- `MS 1..* ← 1..1 MFA`

=== High-Level Structure

For a single frontend channel:


[MFE] → [MFA] → [MS]

For multiple frontend channels (e.g. web, mobile, agentic) sharing the same MS:


[Web MFE]     → [Web MFA]     →
[Mobile MFE]  → [Mobile MFA]  →  [MS]
[Agentic MFE] → [Agentic MFA] →

Each MFE/MFA pair is tightly coupled to one another and to one bounded context, while the MS can serve multiple such pairs.

== Why Not the BFF Pattern?

The classical Backend-for-Frontend (BFF) pattern is designed to:

- Aggregate multiple backend services
- Construct composite view models
- Apply frontend-specific business logic
- Serve as a mini-backend for a UI domain

This is incompatible with:

- DDD bounded contexts
- Hexagonal Architecture
- Clean ubiquitous language
- Incremental delivery
- Micro-frontend autonomy

A BFF becomes a **cross-domain orchestration layer**, which:

- Entangles ubiquitous languages
- Couples multiple bounded contexts into one UI
- Slows down independent delivery
- Introduces long-term maintainability risks
- Violates the principle of domain isolation

The MFA pattern avoids these pitfalls by being:

- Thin
- Context-specific
- Boundary-focused
- Non-aggregating
- Non-orchestrating

== The Micro-Frontend Adapter (MFA) Pattern

The MFA is a **frontend-side adapter** that mirrors the backend’s hexagonal architecture and is **1:1 paired** with its MFE.

=== Responsibilities of the MFA

The MFA has **strictly limited responsibilities**, aligned with boundary concerns:

- Validate the session cookie provided by the MFE
- Exchange the session cookie for a user token required by the MS
- Enforce authentication and authorization boundaries
- Optionally evaluate feature flags
- Perform light request/response shaping:
  - renaming fields
  - flattening structures
  - hiding backend details
- Maintain a stable API contract for the MFE
- Prevent backend API evolution from breaking the UI
- Ensure backend-facing tokens are never exposed to the MFE
- Forward validated and translated requests to the MS

=== Non-Responsibilities of the MFA

The MFA explicitly does *not*:

- Aggregate multiple microservices
- Apply domain logic
- Construct view models
- Perform cross-service orchestration
- Implement workflows
- Become a mini-backend

This prevents “BFF bloat” and keeps the architecture clean.

=== MFA as a Hexagonal Component

The MFA itself follows Hexagonal Architecture and is designed to be developed and evolved in **lockstep** with its corresponding MS:


    +------------------------+
    |   Micro-Frontend       |
    |        (UI)            |
    |      (1..1 with MFA)   |
    +-----------+------------+
                |
                | Port: MFE API
                v
    +------------------------+
    |  Micro-Frontend        |
    |      Adapter (MFA)     |
    | (1..1 with MFE, 1..1   |
    |        with MS)        |
    +-----------+------------+
                |
                | Port: MS API
                v
    +------------------------+
    |     Microservice       |
    |   (Domain + Adapters)  |
    |   (1..* MFAs)          |
    +------------------------+


This symmetry ensures:

- Replaceability
- Testability
- Maintainability
- Independent evolution
- Clear separation of concerns

== The Frontend Anti-Corruption Layer (ACL)

The collection of all MFAs forms the **Frontend Anti-Corruption Layer**.

The ACL ensures:

- The frontend never directly depends on backend domain models
- Backend changes do not leak into the UI
- Ubiquitous language remains pure within each bounded context
- MFEs remain independent and context-specific
- No cross-context contamination occurs

=== ACL Boundaries

The ACL boundary is defined by:

- The public API of each MFA
- The mapping between MFE language and MS domain language
- The contract translation rules
- The authentication and feature-flag boundaries
- The prohibition of cross-context calls

No MFA may call an MS outside its bounded context, and no MFE may call an MS directly.

=== Bounded Context Boundaries

Each bounded context is documented through:

- Its domain model
- Its ubiquitous language
- Its MS API
- Its MFE API (via the MFA)
- Its BDD feature files
- Its architectural diagrams

The **MFE + MFA pair** is scoped to a single bounded context:

- The MFE uses the ubiquitous language of that context.
- The MFA translates between the MFE’s API and the MS’s domain model.
- No MFE or MFA spans multiple bounded contexts.

This prevents entanglement of ubiquitous languages and keeps BDD scenarios clean and domain-aligned.

== Benefits for Engineers, Testers, CTOs, and Auditors

=== For Software Engineers

- Clear 1:1 pairing between MFE and MFA reduces cognitive load.
- MFEs and MSs evolve independently; MFAs absorb contract changes.
- Hexagonal symmetry improves testability and maintainability.
- No accidental cross-context coupling via a shared BFF.

=== For Test Engineers

- BDD scenarios map cleanly to bounded contexts.
- MFAs provide stable, testable contracts per frontend channel.
- No mixed-domain test complexity.
- Easier automation and scenario expansion per MFE/MFA pair.

=== For Skeptics

- MFAs prevent long-term BFF bloat and cross-domain complexity.
- MFAs reduce coupling, not increase it, despite more components.
- MFAs enable faster incremental delivery of features and scenarios.
- The overhead is structural, not accidental, and pays off as the system grows.

=== For CTOs

- Architecture scales with teams, channels, and features.
- Supports parallel delivery streams (web, mobile, agentic, etc.).
- Reduces long-term maintenance cost and risk.
- Aligns with modern DDD, MSA, and Hexagonal practices.
- Provides clean auditability and governance across bounded contexts.

=== For Auditors

- Clear separation of concerns and responsibilities.
- Explicit, documented boundaries and cardinalities.
- Traceable data flow from MFE → MFA → MS.
- No hidden cross-domain dependencies.
- Strong alignment with industry best practices for modular, secure systems.

== Appendix A: ADR — Adoption of the Micro-Frontend Adapter Pattern

=== ADR-001: Use Micro-Frontend Adapters Instead of the BFF Pattern

==== Status
Accepted

==== Context
We require an architecture that:

- Supports incremental and iterative delivery
- Preserves DDD bounded contexts
- Maintains clean ubiquitous language
- Enables independent evolution of MFEs and MSs
- Avoids cross-context coupling
- Aligns with Hexagonal Architecture
- Supports BDD practices
- Supports multiple frontend channels (web, mobile, agentic) per bounded context

The traditional BFF pattern aggregates multiple microservices and introduces cross-context coupling, violating these principles. It also encourages a single component to serve multiple bounded contexts and frontend channels, entangling ubiquitous languages and increasing long-term complexity.

==== Decision
We adopt the **Micro-Frontend Adapter (MFA)** pattern with explicit cardinalities:

- Each MFE associates with one and only one MFA.
- Each MFA associates with one and only one MFE.
- Each MFA associates with one and only one MS.
- Each MS can associate with one or more MFAs (for different frontend channels).

Additional constraints:

- MFA forms part of the Frontend ACL.
- MFA handles boundary concerns only (auth, feature flags, light shaping).
- MFA mirrors Hexagonal Architecture and is developed in lockstep with its MS.
- No aggregation or orchestration across multiple MSs is allowed in an MFA.
- No MFE or MFA spans multiple bounded contexts.

==== Consequences

Positive:

- Clean bounded contexts and ubiquitous languages.
- Independent delivery of MFEs and MSs per bounded context.
- Maintainable long-term architecture as features and channels grow.
- Clear testability and BDD alignment per context and per channel.
- Reduced coupling and complexity compared to a shared BFF.
- Clear, auditable boundaries and data flows.

Negative:

- More deployable components than a single BFF.
- Requires discipline to maintain 1:1 MFE–MFA pairing and context boundaries.
- Initial setup may appear heavier to teams focused only on the first release.

These trade-offs are acceptable and aligned with long-term architectural goals, especially as the number of features, scenarios, and frontend channels increases.
