= Security MFA Example
:toc: left
:toclevels: 2
:icons: font
:source-highlighter: rouge
:doctype: article

A working example of an MFA (Micro-Frontend Adapter) that implements a complete OAuth2 Authorization Code flow, session management, Authorization Service token swap, and API-key validation.

The example uses WireMock to simulate both the OIDC Identity Provider (Keycloak-style paths) and the Authorization Service, so the entire five-service topology runs locally with no external dependencies.

This example is derived from the route-mapping example.
The route-rewriting and field-name-translation behaviours are preserved unchanged.
The additions are:

. An `OidcProviderAdapter` that drives the OAuth2 Authorization Code flow with the OIDC IdP and builds the authorization redirect URL.
. An `AuthenticationController` that exposes `GET /auth/login`, `GET /auth/callback`, and `POST /auth/logout` endpoints.
. An `AuthenticationApplicationService` that exchanges authorization codes for session objects, persists them, and handles logout.
. An `AuthorizationServiceAdapter` that swaps the user token for an inner token enriched with RBAC/ABAC claims.
. A `SecurityMfaProperties` `@ConfigurationProperties` class that carries the routing pattern, OIDC provider coordinates, and session settings from YAML configuration.
. An Angular `authGuard` (functional `CanActivateFn`) that checks for the `mfa-session` cookie before allowing route activation.
. An Angular `apiKeyInterceptor` (functional `HttpInterceptorFn`) that attaches gateway API-key headers to every outbound request.

=== Session Cookie Design

The `mfa-session` cookie is set with `http-only: false` by deliberate design.
Angular reads `document.cookie` in the `authGuard` to determine whether a session is active before activating a protected route.
In a production deployment `http-only: true` would be used and the session check would be performed via a dedicated `/auth/session` API call instead.

== Prerequisites

* Java 21
* Node.js 22 / npm 11
* Docker & Docker Compose (for the containerised run)

== Running Locally (without Docker)

=== 1. Start WireMock Services

Start both WireMock containers individually, or use Docker Compose for them only:

[source,bash]
----
docker compose up wiremock-idp wiremock-authz
----

WireMock IdP starts on `http://localhost:8082`.
WireMock Authz starts on `http://localhost:8083`.

=== 2. Start the Microservice

[source,bash]
----
cd ms
export H2_PASSWORD=""
./gradlew bootRun
----

The MS starts on `http://localhost:8081`.

=== 3. Start the MFA (security-mfa profile)

[source,bash]
----
cd mfa
./gradlew bootRun \
  -Dspring.profiles.active=security-mfa \
  -Dmfe-adapter.oidc-provider.issuer-uri=http://localhost:8082/realms/mfa \
  -Dmfe-adapter.authorization-service.base-url=http://localhost:8083 \
  -Dmfe-adapter.microservice.base-url=http://localhost:8081
----

The MFA starts on `http://localhost:8080`.

=== 4. Start the MFE (dev server)

[source,bash]
----
cd mfe
npm install
npm start
----

The Angular dev server starts on `http://localhost:4200` and proxies API calls to `http://localhost:8080`.

Open `http://localhost:4200` in a browser.
The `authGuard` redirects unauthenticated users to `/auth/login`, which initiates the OAuth2 flow.

== Security Features

=== 1. OAuth2 Authorization Code Flow

The `OidcProviderAdapter` drives the complete Authorization Code flow:

* `buildAuthorizationUrl(callbackUri, state)` вЂ” constructs the IdP redirect URL with `response_type=code`, `client_id`, `scope=openid`, and `state`.
* `exchangeCodeForToken(code, redirectUri)` вЂ” calls `POST /realms/mfa/protocol/openid-connect/token` and returns a `UserToken` (access token + subject extracted from the JWT payload).
* `revokeToken(userToken)` вЂ” calls `POST /realms/mfa/protocol/openid-connect/revoke` on logout.

=== 2. Session Management

The `AuthenticationApplicationService` manages the session lifecycle:

* On callback: creates a `Session` with a UUID session ID, the user token, and an expiry calculated from `ttl-minutes`.
* The `SessionStorePort` SPI persists the session (JPA-backed `SessionEntity` in the default implementation).
* On logout: revokes the token at the IdP and deletes the session.
* `getValidSession(sessionId)` вЂ” validates that the session exists and is not expired; throws `AuthenticationException` otherwise.

=== 3. Authorization Service Token Swap

The `AuthorizationServiceAdapter` calls `POST /authz/token/swap` with the user token and receives an inner token enriched with RBAC/ABAC claims.
The inner token is forwarded in the `Authorization` header to the microservice.

=== 4. API-Key Validation

The `ApiKeyValidator` SPI implementation (registered via `SecurityMfaConfiguration`) checks for the presence and value of the `X-API-Key-Validated: true` header on every inbound request.
The MFE's `apiKeyInterceptor` adds this header (and `X-API-Key-Scope: READ_WRITE`) to every `HttpClient` request, simulating what an API gateway would inject.

The Relationships tab (`RelationshipsService`) deliberately uses the native `fetch()` API instead of `HttpClient`, which bypasses the `apiKeyInterceptor`.
As a result, the MFA receives requests to `/v1/dutch/familyties/relationships/вЂ¦` with no `X-API-Key-Validated` header and immediately returns `403 Forbidden`.
Opening the Relaties tab fires this probe automatically so the forbidden response is visible without any manual form submission.

=== 5. Angular Route Guard

The `authGuard` (`CanActivateFn`) reads `document.cookie` to check for the `mfa-session` cookie before activating any protected route.
If the cookie is absent the guard navigates to `/auth/login`.

== security-mfa Profile вЂ” What Is Activated / Disabled

Under `spring.profiles.active=security-mfa`:

.Activated by `@ConditionalOnProperty`
[cols="1,2", options="header"]
|===
|Bean / Class                       |Condition
|`OidcProviderAdapter`              |`mfe-adapter.session.required=true`
|`AuthenticationController`         |`mfe-adapter.session.required=true`
|`AuthenticationApplicationService` |`mfe-adapter.session.required=true`
|`AuthorizationServiceAdapter`      |`mfe-adapter.authorization-service.required=true`
|===

.Disabled (not on classpath in this example)
* Kafka (`KafkaAutoConfiguration` excluded) вЂ” no reference-data events
* mTLS (`mfe-adapter.mtls.enabled=false`) вЂ” plain HTTP to keep the topology simple
* Spring Security OAuth2 client auto-configuration вЂ” the MFA manages its own OIDC flow
