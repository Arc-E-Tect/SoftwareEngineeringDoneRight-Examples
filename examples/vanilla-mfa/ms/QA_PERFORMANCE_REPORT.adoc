=== QA Pipeline Tasks

The QA task executes the following verification pipeline in order:

. build — Compiles all code and generates Spring REST Docs
. testContractWiremock — Verifies WireMock contract conformance (with WireMock container)
. testSystemWiremock — System tests against WireMock (with WireMock container)
. composeDown — Stops and cleans up WireMock Docker containers
. testContract — Spring REST Docs contract tests (H2 in-memory)
. testSystem — System tests with Testcontainers (PostgreSQL)
. testE2E — End-to-end tests with full Docker Compose stack (app + PostgreSQL)

=== Critical Infrastructure

- WireMock Container: Port 8443 (HTTPS) and 8089 (HTTP)
- E2E Application Container: Port 8443 (HTTPS)
- PostgreSQL Container: Port 5432
- Container Cleanup: Automatic via Docker Compose plugin and explicit `composeDown` task

== Detailed Analysis

=== Why Configuration Cache Is Disabled

The QA task explicitly disables configuration cache because:

. Docker Container Lifecycle: The task depends on proper sequencing of Docker Compose operations (`composeUp` → tests → `composeDown`)
. Port Conflicts: WireMock containers (port 8443) must be fully stopped before E2E tests start their containers on the same port
. Task Up-to-Date Checks: With configuration cache, Gradle aggressively caches task outputs, causing `composeDown` to be skipped when WireMock tests are marked as up-to-date
. Simplicity: Disabling configuration cache keeps the QA task simple and predictable without needing complex workarounds

== Key Findings

=== Docker Container Management

The QA task properly manages Docker containers through task dependencies:

. WireMock Phase: `testContractWiremock` and `testSystemWiremock` use WireMock on port 8443
. Cleanup Phase: `composeDown` stops and removes WireMock containers
. E2E Phase: `testE2E` starts application container on port 8443

----
[.source]
----
build → testContractWiremock → testSystemWiremock → composeDown → testContract → testSystem → testE2E
----

This ensures no port conflicts between WireMock and E2E containers.

=== Configuration Cache Trade-off

Decision: Disable configuration cache for QA task

Reasoning:
- Configuration cache optimizes task execution by caching task outputs
- When WireMock tests are up-to-date, Gradle skips `composeDown`, leaving containers running
- This causes port 8443 conflicts when E2E tests try to start
- Workarounds (outputs.upToDateWhen, doFirst hooks) add complexity
- Performance impact is minimal: configuration cache would save ~2-3 seconds on incremental builds
- Simplicity and reliability trump marginal performance gains

Implementation: Mark the QA task incompatible with configuration cache using `notCompatibleWithConfigurationCache()` in `build.gradle`.

== Appendix: Task Dependencies

=== QA Task Configuration (build.gradle)

[source,gradle]
----
// QA task: runs a full verification pipeline in order
// Note: Configuration cache is not supported for QA task - use --no-configuration-cache
tasks.register('QA') {
    group = 'verification'
    description = 'Full QA: build, verify wiremock, contract, system and e2e tests. Use with --no-configuration-cache'
    dependsOn 'build', 'testContractWiremock', 'testSystemWiremock', 'composeDown', 'testContract', 'testSystem', 'testE2E'
}

afterEvaluate {
    tasks.named('testContractWiremock').configure { mustRunAfter 'build' }
    tasks.named('testSystemWiremock').configure { mustRunAfter 'testContractWiremock' }
    tasks.named('composeDown').configure { mustRunAfter 'testSystemWiremock' }
    tasks.named('testContract').configure { mustRunAfter 'composeDown' }
    tasks.named('testSystem').configure { mustRunAfter 'testContract' }
    tasks.named('testE2E').configure { mustRunAfter 'testSystem' }
}
----
